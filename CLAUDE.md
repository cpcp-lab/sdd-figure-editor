# CLAUDE.md

本ファイルは，Claude Code (claude.ai/code) がリポジトリ内のコードや文書をどのように扱うかのガイドを記述します．

## プロジェクト

このリポジトリはオブジェクト指向に基づき，Java で実装した図形エディタを管理します．
設計ドキュメントでは PlantUML で記述した UML クラス図を積極的に用います．

## ビルド・テスト・実行

プロジェクトは Maven を使用することを想定．以下のコマンドは `pom.xml` が存在することを前提とする．

```bash
# ビルド
mvn compile

# テスト実行 (全件)
mvn test

# 特定テストのみ実行
mvn test -Dtest=ClassName#methodName

# 目視確認テストを自動比較モードで実行 (実装後に必ず実行)
mvn test -Dgroups=visual -DexcludedTestGroups= -Dvisual.auto=true

# 実行可能 JAR の作成
mvn package

# アプリケーションの起動
mvn exec:java -Dexec.mainClass="<メインクラス名>"
```

> pom.xml がまだ存在しない場合は，最初のワークパッケージ開始時に作成する．

## アーキテクチャ

- **GUI フレームワーク**: Java Swing を使用した図形エディタ
    - JavaFX は使用しない．
- **設計方針**: オブジェクト指向設計に基づき，図形の継承・ポリモーフィズムを活用
- **主要コンポーネント (予定)**:
  - 図形モデル層 (`Figure` 階層) : 円・矩形・線分などの図形クラスたち
  - ビュー/キャンバス層: 図形の描画と UI 管理
  - コントローラ層: マウスイベントによる図形操作の制御

詳細な設計は `docs/static-structure.md` (クラス図) と `docs/dynamic-process.md` (状態機械図) を参照．

## 文書

開発中に以下の文書を適宜作成します．

1. `docs/` に永続的な文書を作成．
    - `requirements.md` — プロジェクトの目的，おもな機能などを記述．
    - `domain.md` — 図形や図形作成作業のドメイン分析結果 (クラス図) を記述．実装と関連付いてなくてよい．
    - `static-structure.md` — 実装のクラス仕様やクラス間の関係 (クラス図) を記述．クラス図は必ず以下の3つに分割して記述する: (1) 全体構造 (パッケージ間の関係)，(2) Figure 階層の詳細，(3) Tool 階層の詳細．
    - `dynamic-process.md` — 図形作成作業の動的な過程 (状態機械図) を記述．

2. `.steering/[YYYY-MM-DD]_[name-of-the-package]` に各ワークパッケージについてのステアリング文書を作成．
    - 同じ日に複数のワークパッケージを実施する場合，ファイル名は `.steering/[YYYY-MM-DD]_[i]_[name-of-the-package]` とする．`i` は正数 1,... とする．
    - パッケージにおける要求事項を箇条書き．
    - 要求を達成するための方針を記述．
    - 具体的なタスクを，進捗とともに，箇条書きする．
    - **【重要】各ステアリング文書の作成・改訂には承認が必要．**

### 記法

- 句読点は `，`，`．`
- 記号 `(`，`)`，`:`，`;` は半角にする．全角 `（）：；` は使わない．
- 括弧書きの前後，コロンの後などには半角スペースを入れる
    - 良い例: `図形 (Figure) を描く`，`種類: 線分，矩形`
    - 悪い例: `図形(Figure)を描く`，`図形（Figure）`，`種類:線分`

## 開発プロセス

逐次的に機能を追加・改良していくプロセスをとる．
ワークパッケージごとに実行可能プログラムを作成する．
開発項目についてユニットテストケースを作成し，テストが通るようにする．

0. 【最初の準備】永続的ドキュメントの雛形を用意する．
1. ワークパッケージを繰り返し実施．
    1. ステアリング文書を用意する．
    2. タスクがドメインモデルの変更を要する場合，`domain.md` の記述を適宜変更する．
    3. ステアリング文書に定義したタスクを実施していく．
    4. 実装を `static-structure.md` のクラス図に反映する．
    5. 代表的な動的過程を `dynamic-process.md` に状態機械図，シーケンス図，コミュニケーション図などとして描く．
    6. テストケースを作成し，テストを実施する．
    7. 不具合があれば修正する．開発内容について，目視による確認も実施する．

## 基本方針

1. 勝手に作業方針を決めつけないこと．迷いがある場合，隠さないこと．トレードオフを明確に示すこと．
2. 問題を最小限のコードで解決せよ．推測に基づく要素は一切含まないこと．
3. 必要なものだけを改変すること．自分が試行錯誤して生成したものは自分で片付けること．
4. いつも最初に達成基準を定義し，それが検証されるまで繰り返し作業を実施せよ．
